2. Authentication module — request flows

2.1 Local signup — POST /auth/signup

  1. AuthController validates SignupRequest (@Valid), passes to AuthService.signup.
  2. Email lowercased + trimmed via normaliseEmail. userRepository.existsByEmailIgnoreCase → if true, throw BadRequestException("EMAIL_ALREADY_EXISTS").
  3. Initial username generated server-side: SUBSTRING(local-part, 1, 50) + "_" + first-8-of-fresh-uuid, all lowercased. Backed by unique index ux_users_username_lower on LOWER(username).
  4. Password BCrypt-hashed (strength 12 ≈ 250ms). User saved with authProvider=LOCAL, active=true, mustChangePassword=false, emailVerifiedAt=null.
  5. JwtService.generateAccessToken mints HS256 JWT with claims sub=userId, role, email, conditionally schoolId. TTL from app.jwt.access-token-expiration-minutes (default 15 min).
  6. RefreshTokenService.issueFor generates 48 random bytes (SecureRandom) → URL-safe Base64 → SHA-256 hex hash stored. Plaintext is returned to client only. UA + client IP (X-Forwarded-For honoured) recorded for forensics.
  7. Response 201 CREATED, ApiResponse<AuthResponse> with both tokens + a UserDto projection. mustChangePassword drives SPA redirect to rotation flow when an owner pre-created the row with test123.

2.2 Local login — POST /auth/login

  1. Email normalised, user looked up via findByEmailIgnoreCase. Missing → UnauthorizedException("INVALID_CREDENTIALS") (deliberately the same code as wrong-password to avoid email enumeration).
  2. If authProvider != LOCAL or passwordHash == null → UnauthorizedException("USE_OAUTH_LOGIN") with the actual provider name in the human message.
  3. passwordEncoder.matches(raw, hash). Mismatch → INVALID_CREDENTIALS.
  4. active=false → ACCOUNT_DISABLED.
  5. Issue tokens (same path as signup, step 5–7).

2.3 Google OAuth — POST /auth/google

  1. GoogleOAuthService.verify(idToken) calls Google's verifier (audience = our client id). Failure throws UnauthorizedException("INVALID_GOOGLE_TOKEN"). Verifier returns OAuthVerifier(providerUserId, email, fullName).
  2. Upsert priority 1: findByAuthProviderAndProviderUserId(GOOGLE, sub) — if found, return that user. Refresh fullName if it changed at the provider. Reject if !active.
  3. Priority 2: same email exists but different provider → BadRequestException("EMAIL_USED_WITH_DIFFERENT_PROVIDER"). Same provider but missing providerUserId → backfill it.
  4. Priority 3: brand-new user. Requires signupRole in the request — missing throws BadRequestException("ROLE_REQUIRED"). Email pre-verified (emailVerifiedAt = Instant.now()) because Google asserts ownership.
  5. Issue tokens.

2.4 Refresh — POST /auth/refresh (rotation)

  1. RefreshTokenService.consumeAndRotate(rawToken): SHA-256 hash, lookup, fail with INVALID_REFRESH_TOKEN on miss.
  2. isActive(now): not revoked AND not expired. Reuse detection — if revoked-but-presented, revoke ALL active tokens for that user (repository.revokeAllForUser) and throw REFRESH_TOKEN_REVOKED. Defends against replay after theft.
  3. Mark used token revokedAt=now, return its userId. Caller (AuthService.refresh) re-loads the user, re-checks active, issues a fresh access+refresh pair.

2.5 Logout — POST /auth/logout

Look up by hash, set revokedAt if not already revoked. Idempotent — unknown tokens are silently 204'd (no info leak about which tokens existed).

2.6 JWT request flow

  1. JwtAuthenticationFilter (OncePerRequestFilter) reads Authorization: Bearer <token>.
  2. JwtService.parseAndValidate: signature (HS256, secret ≥ 32 bytes enforced at JwtService construction), requireIssuer("sroadtutor"), exp. JwtException/IllegalArgumentException → UnauthorizedException("INVALID_TOKEN").
  3. Build UsernamePasswordAuthenticationToken(userId, null, [ROLE_<role>]) and set on SecurityContextHolder. Critical: only UnauthorizedException is caught and swallowed — any other exception (NPE, IllegalState) propagates, so real bugs aren't masked as 401s.
  4. SecurityUtil.currentUserId() / currentRole() are the controller-side accessors. Both throw UnauthorizedException("NOT_AUTHENTICATED" / "MISSING_ROLE" / "INVALID_PRINCIPAL") rather than returning null.

3. Phone-number module — request flows

3.1 Owner polymorphism — the central design choice

phone_numbers has four nullable owner FKs (user_id, school_id, instructor_id, student_id) plus a CHECK phone_owner_exactly_one that enforces "exactly one is non-null". Java side: PhoneOwnerType enum (USER/SCHOOL/INSTRUCTOR/STUDENT) is the wire format, never persisted as a column. Q1a: there is no PARENT owner — parents are users with role=PARENT, so their phones use USER. The four findByXxxId repository methods + four findByXxxIdAndPrimaryTrue match this shape one-to-one.

3.2 Create — POST /api/phone-numbers

  1. Controller pulls (role, currentUserId) from SecurityUtil; Bean Validation enforces non-blank country code, 4-char max; non-blank national, 20-char max; label ≤ 40.
  2. Scope check: PhoneScopeChecker.requireWriteScope(role, currentUserId, ownerType, ownerId). The 4 × 4 matrix is implemented with explicit per-role private methods. OWNER scope queries users.school_id for the owner-user and confirms target shares it. INSTRUCTOR/STUDENT/PARENT scopes resolve via PhoneOwnershipLookup (raw SQL on instructors, instructor_schools, students, parent_student).
  3. 5-cap enforcement: ownerPhones(type, id).size() >= 5 → BadRequestException("PHONE_LIMIT_EXCEEDED"). Constant MAX_PHONES_PER_OWNER=5 is in the service file, deliberately visible.
  4. E.164 normalisation: E164Normalizer.normalize(cc, nn). Regex ^[1-9][0-9]{0,2}$ for country (no leading zero), ^[0-9]{4,14}$ for national, total ≤ 15 digits. Returns (cc, nn, "+ccnn"). All failures throw INVALID_PHONE_NUMBER.
  5. Same-owner duplicate check: stream existing phones, fail with PHONE_ALREADY_EXISTS on match. Race-safe authority is the partial unique index ux_phone_<owner>_e164; this check just gives a friendly error pre-DB.
  6. Build entity (whatsapp=true, whatsappOptIn=true by default, primary=false, verifiedAt=null), set the correct owner FK via applyOwner, save.
  7. Primary promotion: shouldBePrimary = makePrimary || existing==0. If true, promoteToPrimaryInternal demotes any existing primary on the same owner (using the owner-specific findByXxxIdAndPrimaryTrue), then promotes the new row. Both saves run in the same @Transactional so the partial unique index never sees two primaries simultaneously.
  8. Response 201 CREATED with PhoneNumberResponse projection — ownerType derived from whichever FK is non-null.

3.3 List — GET /api/phone-numbers?ownerType=&ownerId=

Both query params optional. Default = caller's own user phones (USER + currentUserId) — the dominant SPA call. Scope-checked the same way as create. Read scope == write scope at V1 (PhoneScopeChecker.requireReadScope just delegates to requireWriteScope).

3.4 Get one — GET /api/phone-numbers/{id}

Repository findById; missing → ResourceNotFoundException (404). Scope check derives owner from the entity (requireReadScope(role, userId, phone) overload).

3.5 Update — PUT /api/phone-numbers/{id}

  1. Load + scope check.
  2. Atomic CC/NN rule: countryCode and nationalNumber must travel together. Sending one without the other → BadRequestException("INVALID_PHONE_NUMBER", "countryCode and nationalNumber must be updated together"). Stops a partial update from leaving a corrupted e164.
  3. If both supplied AND new e164 != current → re-check same-owner uniqueness, set the three columns, null out verifiedAt (number changed → previous verification is stale).
  4. Apply nullable label/whatsapp/optIn deltas. Save.

3.6 Promote-primary — POST /api/phone-numbers/{id}/primary

Idempotent (already-primary → return as-is). Otherwise the demote-then-promote dance from §3.2 step 7.

3.7 WhatsApp opt-in toggle — POST /api/phone-numbers/{id}/whatsapp-optin

Pure setter on whatsappOptIn. Useful for the SPA's "Stop WhatsApp" toggle without re-entering the number.

3.8 Delete — DELETE /api/phone-numbers/{id}

Hard delete. ON DELETE CASCADE on whatsapp_message_log.recipient_phone_id wipes audit rows for the deleted phone — see §6 tech-debt for why this matters.

4. WhatsApp module — request flows

4.1 Generate link — POST /api/whatsapp/links

  1. Load recipient phone (findById, 404 on miss). requireReadScope(senderRole, senderUserId, phone) — same gate as phone CRUD.
  2. !whatsapp || !whatsappOptIn → WHATSAPP_DISABLED.
  3. Body resolution: templateId wins. If supplied but row missing → 404. If row exists but !active → TEMPLATE_INACTIVE. Otherwise renderPlaceholders(body, placeholders) substitutes {{key}} tokens via regex \{\{\s*([a-zA-Z0-9_.-]+)\s*}}; missing key → PLACEHOLDER_MISSING. Matcher.quoteReplacement escapes $/\ in values.
  4. If no templateId but body non-blank → trim and use as-is.
  5. If neither supplied (or body is blank/whitespace) → MISSING_BODY_OR_TEMPLATE.
  6. Pull sender's schoolId from users (may be null — column is nullable).
  7. Persist a WhatsappMessageLog row: senderUserId, recipientPhoneId, templateId (nullable), renderedBody (verbatim, audit-grade), schoolId, correlationId. linkGeneratedAt defaults to now via @PrePersist.
  8. Build URL: https://wa.me/<cc+nn>?text=<URLEncoder.encode(body, UTF_8)>. The "+" is intentionally stripped (wa.me spec).
  9. Response 201 CREATED with WaMeLinkResponse(logId, url, generatedAt, …).

4.2 Click confirm — POST /api/whatsapp/links/{logId}/click-confirm

  1. Load log row (findById, 404 on miss).
  2. Sender == clicker auth model: if logRow.senderUserId != currentUserId → AccessDeniedException (403 via GlobalExceptionHandler) with message "Only the link generator can confirm the click". This is the verification model — D14: "trust the typist + WhatsApp click-confirm". Trade-off: cross-device verification (instructor generates, student clicks) doesn't fly. Tracked as TD.
  3. Idempotent click time: if linkClickedAt already set, reuse it; else set to Instant.now() and save.
  4. Reload phone (defensive — could have been deleted between generation and confirm). If phone.verifiedAt == null → set to clickAt, save, mark verifiedNow=true. Subsequent confirms are no-ops on the verification flag.
  5. Response: ClickConfirmResponse(logId, phoneId, clickAt, verifiedNow, phone.verifiedAt).

8. Code-review findings — by severity

P0 · BadRequestException prefix duplicates code in API response
BadRequestException (line 16): super("[" + code + "] " + message). GlobalExceptionHandler.handleBadRequest (lines 38–41) sends both code and message into ErrorResponse. Result: client sees the code in two fields, and the human message has a noisy bracket prefix. UnauthorizedException (line 8) does NOT prefix — so the API is now inconsistent across status codes. Fix: revert super("[" + code + "] " + message) to super(message); update test assertions in PhoneNumberServiceTest (L185, L334) and WaMeServiceTest (L229, L246, L267) from .hasMessageContaining(CODE) to either .extracting("code").isEqualTo(CODE) or a custom BadRequestException::getCode assertion.
P0 · Blocks production
P1 · WaMeService URL has no body-length cap
buildWaMeUrl URL-encodes renderedBody with no length guard. WhatsApp's wa.me silently truncates somewhere around 2 KB; long template bodies (or large placeholder values) will produce links that truncate mid-message. Fix: validate renderedBody.length() < 2000 after rendering; throw BODY_TOO_LONG.
P1 · Pre-launch
P1 · WaMeController.confirmClick has no scope check
By design — sender == clicker check is the auth model. But: if a sender's account is later disabled (active=false), they can still call confirmClick and verify a phone (no active re-check). Low-impact (account disable is rare) but worth a guard. Fix: in confirmClick, also load the sender user and reject if !active.
P1
P1 · whatsapp_message_log cascade-deletes on phone delete
recipient_phone_id UUID NOT NULL REFERENCES phone_numbers(id) ON DELETE CASCADE. Hard-deleting a phone wipes its entire reminder history — bad for tax/audit (D17 says "every wa.me link"). Fix options: (a) soft-delete phones (add deleted_at, change findById to filter); (b) change FK to ON DELETE RESTRICT + force soft-delete; (c) drop the FK to nullable SET NULL. Option (a) is the most invasive but the most correct — the audit log keeps the data forever.
P1
P1 · whatsapp_message_log records absolute body, no PII redaction
renderedBody stores the substituted body verbatim — including {{name}}, {{phone}}, etc. Once school count grows, this column will hold meaningful PII at rest. Fix: column-level encryption or a configurable retention policy (auto-purge log rows > N days). Mention to legal/PIPEDA before launch.
P1 · PIPEDA
P2 · Hand-rolled E.164 vs libphonenumber
E164Normalizer regex passes any digits that fit the shape. +1 0000000000 is accepted (no NA-NPA validation). Acceptable for V1 (controlled SPA inputs); becomes a problem when we open a webhook ingest path or expand to a market with national-trunk-code rules (Italy, Argentina). Fix: swap E164Normalizer internals for libphonenumber. The seam is clean — one class, no caller changes.
P2 · Tracked
P2 · 5-cap is a Java constant, not plan-tier-aware
MAX_PHONES_PER_OWNER=5. Free/Pro/Growth/Enterprise plans will want different caps. Fix: when subscriptions module lands, replace with a SubscriptionService.phoneLimitFor(schoolId) lookup.
P2
P2 · Email enumeration via USE_OAUTH_LOGIN
AuthService.login distinguishes "wrong password" (INVALID_CREDENTIALS) from "this email exists but is OAuth-managed" (USE_OAUTH_LOGIN). An attacker can probe whether a given email is registered at all, and which provider. Mitigated by rate-limiting at the gateway, but the codes themselves are still informative. Fix options: (a) collapse both to INVALID_CREDENTIALS; (b) leave but document. We've kept the discriminating code for usability — confirms with you before we change.
P2 · Decision pending
P2 · No cleanup job for expired refresh tokens
RefreshTokenRepository.deleteExpiredBefore exists but no scheduled task calls it. Table grows monotonically. Fix: @Scheduled(cron="0 0 3 * * *") cleanup bean, or a Postgres pg_cron job.
P2
P2 · No rate-limit on /auth/login or /auth/google
Brute-force protection is currently relying on BCrypt cost (250 ms/hash). Sufficient for Phase 1 traffic; not for any post-launch threat model. Fix: bucket4j or Spring Cloud Gateway rate-limit at the edge. Track per-IP + per-email.
P2
OK · Two saves in promoteToPrimaryInternal are atomic
Caller (create, promoteToPrimary) is @Transactional; the demote-then-promote pair runs inside a single tx, so the partial unique index ux_phone_primary_* never sees two primaries simultaneously even on concurrent calls. Verified.
Verified
OK · JWT claim hygiene
schoolId claim only emitted when non-null (avoids ambiguity between "" and null). HS256 secret length validated at construction (≥ 32 bytes). Issuer required on parse. Defensive.
Verified
OK · Refresh-token reuse detection
Presenting an already-revoked token wipes ALL the user's active tokens (revokeAllForUser), not just the presented one. This is the right play — assume theft and force a clean re-login.
Verified
OK · ddl-auto=validate on all profiles
No create or update auto-DDL anywhere. Flyway is the only schema author. Mismatch between @Entity and DB → app refuses to boot. This is the right strictness.
Verified
OK · No PII in INFO logs
All service-layer info logs use UUIDs, role names, owner-types — never raw email or phone. Critical for PIPEDA when audit retention lands.
Verified
OK · OAuth ID-token verification is canonical
GoogleOAuthService uses Google's official GoogleIdTokenVerifier with the configured client id as audience. No JWT-decoding shortcuts.
