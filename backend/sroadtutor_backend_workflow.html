4. School module — request flows NEW

4.1 Create school — POST /api/schools

  1. SchoolController.create resolves (role, currentUserId) from SecurityUtil; Bean Validation enforces non-blank name, jurisdiction regex, 2-letter province.
  2. SchoolService.createForCurrentOwner:
    • Rejects non-OWNER role → NOT_OWNER.
    • Loads the caller User; ResourceNotFoundException if missing.
    • Two stacked guards: schoolRepo.existsByOwnerId → ALREADY_OWN_SCHOOL; caller.schoolId != null → USER_ALREADY_IN_SCHOOL.
    • Builds the School with planTier=FREE, active=true, synthetic=false, metadata="{}"; tax IDs are nullIfBlank-ed.
    • Persists the school, then sets caller.schoolId = school.id and saves the user (back-link, same transaction).
  3. Response 201 CREATED with full SchoolResponse.

4.2 Read flows

EndpointBehavior
GET /api/schools/meAny authed user. Resolves caller.schoolId; 404 if user is unaffiliated. Returns lean SchoolMeResponse (no tax IDs, no Stripe customer id).
GET /api/schools/{id}OWNER of THIS school only. Non-owner role → 403; OWNER of different school → 403. Returns full SchoolResponse.

4.3 Update — PUT /api/schools/{id}

  1. Load + requireOwnerOf scope check.
  2. Refuses if school.isActive() == false → SCHOOL_INACTIVE (force reactivate first).
  3. Patches name / jurisdiction / province if non-null. Tax IDs accept blank-as-clear semantics: empty string nulls the field, otherwise trim + persist.
  4. Returns refreshed SchoolResponse.

4.4 Lifecycle — POST /api/schools/{id}/deactivate & /reactivate

Both idempotent. Hard delete is intentionally not exposed — instructor / student / session / audit rows still reference this id, and we never want history to disappear.

5. Instructor module — request flows NEW

5.1 Self-register — POST /api/instructors/me

  1. Caller must be authenticated user with role=INSTRUCTOR (else NOT_INSTRUCTOR).
  2. existsByUserId guard → INSTRUCTOR_ALREADY_EXISTS.
  3. Builds Instructor with all profile fields nullable except userId. WorkingHours serialized to JSONB via toJson().
  4. No school attachment yet. The user's schoolId stays null until an OWNER attaches them.

5.2 Profile reads — GET /api/instructors/me + GET /api/instructors/{id}

By-id read scope: instructor themselves, OR an OWNER of any school they're currently attached to (via instructor_schools with leftAt IS NULL). Cross-school OWNERs cannot read each other's instructor rosters.

5.3 Update — PUT /api/instructors/{id}

  1. Load + requireWriteScope (self OR attached-school OWNER).
  2. Inactive profile → INSTRUCTOR_INACTIVE.
  3. Partial-patch semantics on every field; workingHours re-serialized to JSON if supplied.

5.4 Attach instructor to school — POST /api/schools/{schoolId}/instructors/{instructorId}/attach

  1. requireOwnerOfSchool on the target school. Inactive school → SCHOOL_INACTIVE; inactive instructor → INSTRUCTOR_INACTIVE.
  2. Loads existing InstructorSchool by composite PK. Three branches:
    • None → create new link with roleAtSchool (defaults to REGULAR) and joinedAt = now.
    • Active link exists (leftAt IS NULL) → ALREADY_ATTACHED.
    • Old left link → re-hydrate: clear leftAt, refresh joinedAt + roleAtSchool. Audit history preserved by reusing the row.
  3. Back-link: if the instructor's User has no schoolId, set it. Likewise back-fill the legacy instructors.school_id if blank — keeps queries that haven't migrated to the M:N join still pointing at a sensible default.

5.5 Detach — POST /api/schools/{schoolId}/instructors/{instructorId}/detach

Sets leftAt = now on the join row. Idempotent — already-detached returns the row unchanged. The instructor's profile is NOT deactivated; they remain available for attachment to other schools.

5.6 List for school — GET /api/schools/{schoolId}/instructors

OWNER of school OR any INSTRUCTOR currently attached. Students/parents do not get the roster. Returns active rows via the EXISTS+legacy fallback JPQL in InstructorRepository.findActiveBySchool.

6. Student module — request flows NEW

6.1 Add student — POST /api/schools/{schoolId}/students

The marquee endpoint. Atomic: User row + Student row + (optional) Parent User + Parent link, all in one @Transactional.
  1. Load school; SCHOOL_INACTIVE if not active.
  2. requireOwnerOrInstructorOfSchool — OWNER of school OR an INSTRUCTOR with instructors.school_id matching. (See TD-13: legacy-only check; multi-school instructors on non-primary school may be denied. V1 acceptable.)
  3. Email collision guard → STUDENT_EMAIL_ALREADY_EXISTS.
  4. If instructorId supplied: load and verify instructor.schoolId == this schoolId → INSTRUCTOR_WRONG_SCHOOL.
  5. Build student User: lowercased email, generated username (mirrors AuthService.generateInitialUsername shape — local-part + 8 hex), BCrypt.encode("test123"), mustChangePassword=true, role=STUDENT, authProvider=LOCAL, schoolId set, active=true.
  6. Save user. Build Student with packageTotalLessons (default 0), lessonsRemaining (defaults to total if not supplied; refused if > total with REMAINING_GT_TOTAL), status=ACTIVE.
  7. Save student. If parentEmail supplied:
    • findOrCreateParent: lookup by email. Found AND role=PARENT → reuse. Found AND any other role → PARENT_EMAIL_BELONGS_TO_OTHER_ROLE. Not found → create PARENT user with the same dummy-password pattern.
    • Save ParentStudent link with relationship (defaults to "PARENT").
  8. Response 201 CREATED with full StudentResponse (parents inline).

6.2 Read flows

EndpointBehavior
GET /api/students/meCaller's own student profile (role=STUDENT). 404 if no profile.
GET /api/students/{id}Read scope: OWNER of student's school, INSTRUCTOR assigned to this student, the student themselves, or a linked PARENT. 403 otherwise.
GET /api/schools/{schoolId}/studentsOWNER of school OR INSTRUCTOR at school. INSTRUCTOR sees only their assigned students; OWNER sees the full roster.

6.3 Update — PUT /api/students/{id}

  1. Load + requireWriteScope (OWNER of school OR assigned INSTRUCTOR).
  2. If instructorId supplied: cross-school check → INSTRUCTOR_WRONG_SCHOOL.
  3. Apply nullable patch fields. After applying, re-check lessonsRemaining <= packageTotalLessons → REMAINING_GT_TOTAL.
  4. Returns refreshed StudentResponse with parent links populated.

6.4 Parent linkage — POST /api/students/{id}/parents & DELETE /.../{parentUserId}

Same find-or-create-by-email pattern as the add-student parent linkage. Pair-uniqueness enforced both at the DB (uk_parent_student_pair) and pre-checked at the service (PARENT_ALREADY_LINKED). Unlink deletes the join row only — never the parent User itself.

12. Architecture decisions added this PR

ADR-11 · One school per owner (V1)
Enforced at service layer with two guards (existsByOwnerId + users.schoolId != null). DB has no unique index — easier to relax in V2 if owners legitimately run multiple schools. Rejected: DB-level enforcement (premature).
ADR
ADR-12 · Soft-delete everywhere
Schools / instructors / students / instructor_schools all carry status flags or boolean. No hard-delete endpoints. Audit + downstream FKs (sessions, payments, reminders) stay intact even when an entity is "removed".
ADR
ADR-13 · Atomic add-student creates 1..3 rows
StudentService.addByOwner creates User + Student + (optional) Parent User + Parent link in one transaction. Single endpoint, single rollback boundary. Rejected: separate "create student" → "link parent" two-step flow (introduces orphan-row window).
ADR
ADR-14 · Find-or-create-parent by email
Parent linkage looks up by email; existing PARENT → reuse, missing → create with dummy password, existing non-PARENT → refuse. Rejected: silent role mutation (bad UX), separate "first sign up parent" step (worse UX for owner).
ADR
ADR-15 · WorkingHours typed wrapper around JSONB
DB stays JSONB so we never schema-migrate when the model evolves. Java-side WorkingHours record provides typed access via Jackson JavaTimeModule. Rejected: separate working_hours + time_ranges tables (overkill for V1).
ADR
ADR-16 · Instructor M:N rehydration on re-attach
Re-attaching a previously-detached instructor clears leftAt on the existing row rather than inserting a new one. Audit history is preserved through the (instructor_id, school_id) primary key. Rejected: insert-new-row-each-time (loses tenure summation, breaks PK).
ADR
