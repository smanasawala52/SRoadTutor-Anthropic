14. PR8 — Email verification module NEW

14.1 Issue token — POST /auth/email-verify/send

  1. Resolves caller via SecurityUtil.currentUserId() — JWT-required.
  2. EmailVerificationService.issueForCurrentUser: loads User, refuses if already verified (ALREADY_VERIFIED).
  3. Revoke prior actives first: every unconsumed token for this user is marked consumedAt=now before a new one is issued. Maintains the "at most one active per user" invariant. The response carries a reissued boolean so the SPA can render a different toast for first-issue vs re-issue.
  4. Generates 48 random bytes (SecureRandom), URL-safe Base64-encoded → raw token. SHA-256 hex of that goes into token_hash.
  5. Persists the EmailVerificationToken with expiresAt = now + 24h.
  6. Logs the dev verify URL at INFO level. Returns { tokenId, rawToken, verifyUrlForDev, expiresAt, reissued }.

14.2 Confirm — POST /auth/email-verify/{token}/confirm (PUBLIC)

  1. SHA-256 hashes the raw token, looks it up. Missing → INVALID_VERIFY_TOKEN.
  2. isActive(now) guard — expired or already-consumed → VERIFY_TOKEN_EXPIRED_OR_USED.
  3. Loads the User. If already verified via another path — marks the token consumed (defensive — stops future reuse) and throws ALREADY_VERIFIED.
  4. Sets token.consumedAt = now, user.emailVerifiedAt = now. Both saved in same Tx.
  5. Returns { userId, email, emailVerifiedAt }.

15. PR8 — Invitation module NEW

15.1 Two delivery modes

TOKENRandom URL-safe token (48 bytes Base64), SHA-256 hashed at rest. Raw token returned ONCE in the response (and logged for dev). Status = PENDING. No User row yet.Invitee POSTs /{token}/accept with chosen password → User + role-specific row + (optional) linkages all created in one Tx. Status flips to ACCEPTED.

TypeEndpointAuth scopePre-populates

15.3 Accept flow — POST /api/invitations/{token}/accept (PUBLIC)

  1. Hash + lookup. Missing → INVALID_INVITATION_TOKEN.
  2. Status guards: REVOKED → INVITATION_REVOKED; ACCEPTED → INVITATION_ALREADY_ACCEPTED; expired → lazy-expire (set status to EXPIRED in the same Tx) and throw INVITATION_EXPIRED.
  3. Wrong delivery mode (DUMMY_PWD trying to accept via TOKEN endpoint) → WRONG_DELIVERY_MODE.
  4. Race guard — email collision with an account created since invite issuance → EMAIL_ALREADY_HAS_ACCOUNT.
  5. Create User (BCrypt the supplied password, mustChangePassword=false — they chose it themselves). applyRoleSpecificMetadata dispatches by inv.role:
    • INSTRUCTOR → Instructor + InstructorSchool join from metadata.
    • STUDENT → Student + (optional) parent find-or-create + ParentStudent.
    • PARENT → ParentStudent rows for each studentIds entry.
  6. Set inv.status=ACCEPTED, acceptedAt=now, acceptedUserId=newUser.id. Save invite.

15.4 Lookup — GET /api/invitations/lookup/{token} (PUBLIC)

Pre-accept landing page. Returns minimal info (email, fullName, role, deliveryMode, status, schoolName) so the SPA can render "Welcome — you're being invited as INSTRUCTOR at ABC Driving" before the invitee enters a password. Same status guards as accept; expired tokens are lazy-marked.

15.5 Revoke + reissue (OWNER who issued)

EndpointBehavior
POST /api/invitations/{id}/revokeIdempotent. Status flips to REVOKED. INVITATION_ALREADY_ACCEPTED if status is already ACCEPTED.
POST /api/invitations/{id}/reissueTOKEN-mode only (WRONG_DELIVERY_MODE otherwise). Generates new raw token + hash. Resets expiresAt to now+7d. Status flipped back to PENDING (handles EXPIRED case naturally).
GET /api/schools/{id}/invitations?status=PENDINGOWNER's outstanding-invites view. Status filter optional.

15.6 Metadata serialisation

All optional pre-populate fields are written into invitations.metadata JSONB via Jackson (with JavaTimeModule). On accept, the typed extraction is permissive: missing keys → null; type mismatches → null (best-effort, no throw). This keeps the schema flexible for future invite payloads without migrations.

16.2 Status state machine

SCHEDULED ──┬──> COMPLETED   (decrement student.lessons_remaining)
            ├──> CANCELLED   (no decrement; cancelled_at + cancelled_by populated)
            └──> NO_SHOW     (decrement, allowed even at zero remaining)
SCHEDULED ──> SCHEDULED       (reschedule = update in place; conflict + hours re-checked)
non-SCHEDULED has no transitions (terminal)

16.3 Book — POST /api/sessions

  1. Past-datetime guard → PAST_DATETIME.
  2. Duration validation (15..360 min) → INVALID_DURATION.
  3. Load instructor + student. Active checks (INSTRUCTOR_INACTIVE, STUDENT_NOT_ACTIVE). Cross-school check (INSTRUCTOR_WRONG_SCHOOL).
  4. Load school. Active check (SCHOOL_INACTIVE).
  5. Booking-scope check by role: OWNER of school OR instructor-as-self OR student-as-self.
  6. forceOutsideHours=true permitted for OWNER only (FORCE_REQUIRES_OWNER).
  7. Working-hours validation (skipped if forced):
    • Parse instructor's working_hours_json.
    • Convert scheduledAt + endAt to school's local ZoneId.
    • Refuse if windows span midnight (OUTSIDE_WORKING_HOURS).
    • Look up the weekday's TimeRange list. Must find one that contains both fromTime AND toTime. (S3 — end-time must also be within window.)
  8. Conflict detection (requireNoConflicts):
    • Fetch active sessions for the instructor in window [from − 6h, to) (padding covers the longest possible existing session).
    • For each, half-open overlap test start < to AND end > from. Hit → INSTRUCTOR_BUSY.
    • Same for the student → STUDENT_BUSY.
    • Cancelled / no-show rows excluded by repository query (status filter).
  9. Persist new LessonSession with status=SCHEDULED, createdByUserId=currentUserId.

16.4 Reschedule — PUT /api/sessions/{id}

Update-in-place semantics. Only SCHEDULED → SCHEDULED transitions allowed (SESSION_NOT_RESCHEDULABLE otherwise). Resolves the target tuple from optional fields, falling back to current values. Re-runs conflict check excluding the session being rescheduled (so a session can move within its own time range without colliding with itself).

18. New ADRs

ADR-17 · Token-in-response for V1
Until SMTP infra is wired, both email-verify and invitation accept tokens are returned in the API response (raw token + dev URL) and logged. When email lands, those response fields are stripped — public contract stays the same. Trade-off: dev convenience wins over "tokens never on the wire" purity.
ADR
ADR-18 · DUMMY_PWD invitations auto-accept at create
When deliveryMode=DUMMY_PWD, the User + role-specific row + linkages are created INLINE at invite-create time and the invitation row is marked ACCEPTED immediately. Rejected: keep status=PENDING until first login. Reasoning: the User exists; "pending" would be a lie. mustChangePassword=true on the User is the canonical "you still have something to do" flag.
ADR
ADR-19 · Lazy-expire on read
Invitations don't have a scheduled job to flip PENDING → EXPIRED. Instead, the first read after expiry detects it and updates the row in the same Tx. Trade-off: an "expired but unread" invite stays PENDING in the DB until someone touches it. Acceptable because OWNER's listing endpoint reads everything they care about and would trigger the flip.
ADR
ADR-20 · Conflict detection in Java, not SQL
JPQL has no portable interval-add for Instant + minutes, and a custom Hibernate function would tie us to PostgreSQL. Service narrows the candidate set with a simple scheduledAt range query (covered by partial indexes) then runs the half-open overlap test in Java. Trade-off: can't push the entire check into a single round-trip.
ADR
ADR-21 · Working hours stored as LocalTime, school timezone interprets them
Instructor's working_hours_json is wall-clock (LocalTime) with no offset. Booking validates by converting scheduledAt to the school's ZoneId and comparing. New schools.timezone column (V10) defaults to America/Regina; OWNER can change. Rejected: storing windows as ZonedDateTime — DST transitions would shift the window twice a year, which isn't what an instructor means by "I work 9–5".
ADR
ADR-22 · No-show decrements lessons_remaining
No-show counts as a consumed lesson — instructor showed up. Allowed even at 0 remaining (avoids a deadlock where the instructor can't record the no-show because the package ran dry). Cancel does NOT decrement; the lesson is freed.
ADR
